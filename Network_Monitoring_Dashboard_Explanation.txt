Network Monitoring Dashboard - Explanation & Overview

The Network Monitoring Dashboard (implemented in `SystemFailureMonitor.jsx`) is a real-time tracking tool for JsSIP network reliability. It aggregates data from multiple sources (like Local Storage events via the BroadcastChannel API) to provide a unified view of the application's connection health, WebSocket reliability, and overall network performance.

Here is a breakdown of the main components and metrics tracked:

1. KEY STATUS INDICATORS (Top Cards)
These represent the real-time condition of core communication layers:
- WebSocket Connection: Shows if the socket connection to the telephony server is 'healthy', 'warning', or 'critical', and displays its state (e.g., OPEN, CONNECTING, CLOSED).
- SIP Registration: Monitors if the User Agent (UA) is properly registered with the SIP server and shows its expiry time.
- UA Transport Layer: Indicates the physical transport layer connectivity state (Connected vs Disconnected).
- Keep-Alive Messages: Tracks the time elapsed since the last keep-alive heartbeat message from the server to ensure the connection hasn't silently died.
- Network Quality: Assesses the client's network condition using metrics like RTT (Round Trip Time), Effective Type (e.g., 4g, 3g), and signal strength. Also includes direct Packet Time measurements:
  * TS Offset: The one-way transfer time (latency) calculated as the delta between the backend timestamp and the frontend receive timestamp.
  * Arrival Int: The gap between consecutive backend timestamps to monitor upstream network delay or server lag (should map tightly to the 5-second polling interval).
- System Timeouts: Counts the recent timeout events. High timeout counts indicate instability.
- ICMP Latency (Ping): Tracks the round-trip latency (ping) between the client and the origin server using lightweight periodic requests, allowing direct insight into local internet stability.
  * Technical Implementation:
    - HTTP HEAD Request: Since web browsers restrict native low-level ICMP network packets for security reasons, it emulates the behavior using the Javascript HTTP `fetch` API.
    - Every 5 seconds, it sends a `HEAD` request to the web server root (`window.location.origin`). This is lightweight—it asks the server for the headers but not the body content.
    - It bypasses the browser cache by appending the `{ cache: 'no-store' }` flag so the browser makes an actual round-trip to the network instead of serving a localized memory file.
    - It uses `performance.now()` before and after the request resolves. The difference between those two times is the true server round-trip latency in milliseconds.
    - Timestamp Difference Calculation (Network Time): Network metrics can also be triangulated by comparing packet timestamps. 
        * Packet 1: To calculate the one-way network transfer time, compare the timestamp generated by the backend when the packet was created against the frontend timestamp when the packet was received.
        * Packet 2: To detect packet drops or server lag, compare the current backend timestamp against the previous packet's backend timestamp. Since polling occurs on a strict interval (e.g. 5 seconds), this difference should closely match the interval (5 seconds). Deviations indicate server instability or upstream network delays.
    - Fallback RTT (Network Information API): If the server doesn't respond to the `HEAD` request (meaning the server went offline or the ping dropped), it immediately falls back to invoking `navigator.connection.rtt`. `navigator.connection` is a native Javascript Browser API that returns the connection's estimated round-trip-time (RTT). The browser implicitly calculates this value in the background based on active TCP connections.
2. SYSTEM HEALTH SCORE
A calculated percentage score (e.g., 95%) determining the overall health index based on the stability of WebSockets, SIP Registration, and Network Speed.

3. SUMMARY STATS (Event Counts)
The dashboard keeps historical logs of system conditions, grouped into:
- Live Entries: Current snapshot metrics.
- Success Events: Instances where a critical component fully succeeded.
- Warning Events: Suboptimal conditions that aren't critical failures yet (e.g., Fair network quality).
- Failure Events: Complete connection drops or component failures (critical).

4. DETAILED DATA TABLE
Provides a deep-dive, historical view of every network event with columns for:
- Timestamp: When the event occurred.
- Event Type & Network Component: Which piece of the architecture was affected (e.g., Live Status of SIP Registration).
- Status & Severity: Categorized as healthy, warning, success, info, or critical.
- Description: Human-readable message explaining the status.
- Response Time: The average latency or response time in milliseconds.

5. DASHBOARD CONTROLS
- Pause/Resume Auto-Refresh: "Freeze" the live data to investigate an issue without the dashboard constantly jumping.
- Refresh Rate Control: Dropdown to modify how quickly the UI polls the LocalStorage data (e.g., every 1s, 2s, 5s).
- View Modes (Live / History): Toggle between current diagnostics and past events.
- Filters: Filter by severity (Critical, Warning, Success, Info).
- Export Data: Download the current and historical monitoring logs as a `.json` file for deeper debugging.
- Clear Data: Flush the local monitoring storage logs.

SUMMARY
This dashboard is designed to instantly notify agents and administrators of exactly where a communication failure is happening (e.g., weak local WiFi vs. a dropped WebSocket vs. an expired SIP Registration), reducing downtime and accelerating troubleshooting.
