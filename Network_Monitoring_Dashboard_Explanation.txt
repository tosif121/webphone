Network Monitoring Dashboard - Explanation & Overview

Network Monitoring Dashboard (`SystemFailureMonitor.jsx` mein banaya gaya) ek 100% event-driven, real-time tracking tool hai JsSIP network ki reliability measure karne ke liye. Yeh architecture mock data ya slow polling intervals par rely nahi karta. Iski jagah, yeh strictly native browser APIs (navigator.connection) aur real JsSIP backend triggers (jaise ua.on('connected')) ka use karta hai. Iska sync mechanism `localStorage` aur custom `StorageEvent` dispatching ka istemaal karta hai taaki data update hote hi 0-latency ke saath dashboard repaint ho sake (same-tab aur cross-tab dono mein).

Neeche main components aur track kiye gaye metrics ki detailed explanation hai:

1. KEY STATUS INDICATORS (Network Component Status)
Yeh core communication layers ki real-time condition dikhate hain, jisse exactly pata chalta hai ki poore communication stack mein failure kahan ho raha hai:

- WebSocket Connection: Yeh batata hai ki kya telephony server ke saath physical socket connection ban gaya hai ya nahi. Status mein 'healthy', 'warning', ya 'critical' aata hai, aur iski state dikhti hai (e.g., OPEN, CONNECTING, CLOSED).
  * Reason for Critical: Server down hai, firewall wss:// traffic block kar rahi hai, ya DNS resolve fail ho gaya hai. Agar yeh down hai, toh absolutely kuch bhi aur kaam nahi karega.

- SIP Registration: Agar WebSocket open bhi hai, toh yeh monitor karta hai ki kya UserAgent properly authenticated aur registered hai SIP server ke saath calls karne ke liye. Yeh iska expiry time bhi dikhata hai.
  * Reason for Critical: Invalid password, wrong campaign credentials, ya phir Asterisk/SIP backend ne user ki registration reject kar di hai.

- UA Transport Layer: Yeh `jssip` WebRTC engine ki physical transport layer connectivity state batata hai (Connected vs Disconnected). Yeh monitor karta hai ki kya yeh WebRTC data successfully transmit kar pa raha hai dono taraf.
  * Reason for Critical: Browser ne WebRTC pipe cut kar di hai (aksar tab hota hai jab laptop sleep pe chala jaye, mobile browser background tabs ko throttle kare, ya IceCandidates exchange hone mein fail ho jayein).

- Keep-Alive Messages: Yeh server se aaye last keep-alive heartbeat message ke baad beeta hua time track karta hai (exact background timestamps use karke). Ek healthy status ke liye zaruri hai ki message pichle 14 seconds ke andar aaya ho. (Kaise calculate hota hai: `Date.now() - lastMessage.messageTime < 14000`)
  * Reason for Critical: SIP server silently lag kar raha hai, packets drop kar raha hai, ya overloaded hai. Physical socket shayad "Open" dikhaye, par server actually respond nahi kar raha hota.

- Network Quality: Yeh check karta hai ki kya issue agent ke weak local WiFi ya poor 4G connection ki wajah se hai, isse pehle ki data server tak pahuche. Yeh metrics use karta hai jaise RTT (Round Trip Time), Effective Type (e.g., 4g, 3g), aur signal strength.
  * Reason for Critical: Agent shayad apne router se physically bohot door hai, kisi degraded 3G cellular network par hai, ya badi files download kar raha hai jo bandwidth ko consume kar rahi hain.
  * Additional Packet Time metrics:
    - TS Offset: One-way transfer time (latency) jo backend timestamp aur frontend ke receive timestamp ke beech ka difference calculate karke nikalta hai. (Kaise banaya: `frontendReceiveTime - backendTime`)
    - Arrival Int: Lagatar aane wale backend timestamps ke beech ka gap. Agar yeh strict 5s polling interval se alag aata hai, toh yeh directly server lag ya upstream network congestion ki taraf ishara karta hai.

- System Timeouts: Yeh recent timeout events ko count karta hai.
  * Reason for Critical: Agent ke browser ne API/SIP requests bhejne ki koshish ki, par server ne time par respond nahi kiya. Yeh severe intermittent internet drops ya server overload ko indicate karta hai.

- ICMP Latency (Ping): Yeh client aur origin server ke beech ka round-trip latency (ping) track karta hai lightweight periodic HTTP requests ka use karke, jisse local internet stability ka direct insight milta hai.
  * Reason for Critical/Warning: Agar ping times 250ms se zyada hain toh yeh indicate karta hai severe network congestion, VPN interference, ya agent ke ISP aur aapke data center ke beech routing issues.
  * Technical Implementation & Calculation:
    - Browser Security Bypass: Modern web browsers native low-level ICMP network packets ko block kar dete hain. Isliye, hum Javascript HTTP `fetch` API se ek "fake ping" emulate karte hain.
    - True Bypass Protocol (no-cors): Ping request ko origin CORS limits se block hone se bachane ke liye hum `mode: 'no-cors'` use kar rahe hain aur `?_cb=random` query pass kar rahe hain cache bust karne ke liye. Yeh TCP/TLS time accurate measure karne mein help karta hai jisse ICMP latency kafi natural (~50ms - 80ms) ati hai.
    - Stopwatch Technique (`performance.now()`): Hum request bhejne se theek pehle `const start = performance.now();` start krte hain. Yeh ek high-precision timer hai.
    - HTTP HEAD Request: Har 5 seconds mein, ek `HEAD` request origin network `esamwad` URL par bheji jati hai. Yeh fast hoti hai kyunki yeh sirf headers check karti hai, poori data body load nahi karti.
    - No Cache (`cache: 'no-store'`): Yeh flag ensure karta hai ki browser seedha network pe jaye aur internal browser cache ko bypass kare.
    - Latency Calculation: Jaise hi HTTP handshake complete hota hai, hum `duration = performance.now() - start` karte hain. Is duration (in milliseconds) ko as-is ICMP Ping Latency consider kiya jata hai.
    - Fallback RTT (Network Information API): Ek `try-catch` block use hota hai. Agar TCP handshake server offline hone ki wajah se poora drop ho jaye, toh script `navigator.connection.rtt` API value plot karta hai as a graceful fallback.
      * Passive vs Active (Ping vs RTT): Ping actively naye request packets (`fetch`) bhejta hai ek instantaneous "right now" exact measurement lane ke liye. Jabki RTT passively browser background mein calculate karta hai.
      * Security Rounding in RTT: Browser intentionally is RTT value ko nearest 25 milliseconds ke multiple (e.g. 50, 75, 100) par round off kar deta hai taaki malicious websites se user tracking/fingerprinting ko roka ja sake.

2. SYSTEM HEALTH SCORE
Yeh ek calculated percentage score hai (e.g., 95%) jo WebSockets, SIP Registration, aur Network Speed ki stability ke basis par overall health index determine karta hai.

3. REAL-TIME EVENT LOGS & SUMMARY STATS
Yeh dashboard completely dynamic true events ke historical logs rakhta hai, jo freely in components mein grouped hain:
- Info Events: Standard system updates and state changes (jaise "Connecting" ya "Waiting for UA").
- Success Events: Wo instances jahan ek critical component puri tarah succeed hua (jaise "Registered").
- Warning Events: Suboptimal conditions ya non-fatal drops (jaise "Timeout Detected" ya "Keep-Alive Heartbeat Missed").
- Failure Events: Critical connection drops ya complete network failures.

4. DETAILED DATA TABLE
Yeh har network event ka ek deep-dive, historical view provide karta hai in columns ke sath:
- Timestamp: Event kab hua.
- Event Type & Network Component: Architecture ka kaunsa hissa affect hua (e.g., Live Status of SIP Registration).
- Status & Severity: Categorized as healthy, warning, success, info, ya critical.
- Description: Status ko samjhane wala aasan (human-readable) message.
- Response Time: Average latency ya response time milliseconds mein.

5. DASHBOARD CONTROLS
- Pause/Resume Auto-Refresh: Live data ko "Freeze" karna taaki dashboard ke hile bina log records aur UI snapshot ko investigate kiya ja sake.
- Dynamic Zero-Latency Repaint: Dashboard data ke aate hi update hota hai, meaning UI kisi timer based polling (jaise har 5s mein refresh) par dependent nahi hai. Jab internal `flushToStorage` execute hota hai (`safeWrite`), ek event propagate ho kar direct React hooks ko update karta hai.
- Chart Type Selector: Network Latency Graph ko "Line Chart" (default) ya "Bar Chart" format mein toggle karne ke liye option deta hai visual debugging improve karne ke liye.
- View Modes (Live / History): Current diagnostics aur past events ke beech toggle karna.
- Filters: Severity (Critical, Warning, Success, Info) ke hisaab se unko filter karna.
- Export Data: Download current aur historical monitoring logs ek `.json` file mein deeper debugging ke liye.
- Clear Data: Local monitoring storage logs ko flush/clear karna.

SUMMARY
Yeh dashboard isliye design kiya gaya hai taaki agents aur administrators ko immediately notify kiya ja sake ki exactly communication failure kahan ho raha hai (e.g., weak local WiFi vs. ek dropped WebSocket vs. ek expired SIP Registration), jisse downtime reduce ho aur troubleshooting tez ho sake.
